# 01 — 에이전트 기초와 PAIS 2.1 지형

[← 목차로](../README.md)

에이전트를 만들기 전에 두 가지를 분명히 합니다. 첫째, "에이전트"가 단일 모델 호출이나 고정된 RAG와 어떻게 다른가. 둘째, PAIS 2.1의 어떤 모듈이 그 차이를 떠받치는가. 이 문서는 경계를 긋고, 모듈 지형을 펼친 뒤, 에이전트 하나가 어떤 구성요소로 이뤄지는지를 해부합니다.

> 본 문서의 수치·동작은 작성 시점(2026-06) VCF 9.1 / PAIF 9.1 / PAIS 2.1 기준이며, 적용 전 공식 문서로 재확인하시기 바랍니다.

---

## 1.1 에이전트 vs RAG vs 워크플로우

세 가지는 자주 섞여 쓰이지만 **결정 주체**가 다릅니다.

- **단일 호출** — 입력 한 번, 출력 한 번. 모델은 받은 컨텍스트 안에서만 답합니다.
- **RAG(검색 증강)** — 답하기 전에 지식베이스를 검색해 관련 문서를 컨텍스트로 넣습니다. 단계는 "검색 → 생성"으로 **고정**돼 있습니다. 무엇을 검색할지, 검색을 할지 말지는 사람이 정한 흐름을 따릅니다. (④의 주제)
- **워크플로우** — 여러 단계를 사람이 미리 정한 순서대로 잇습니다. 분기·반복이 있어도 **경로가 코드로 고정**돼 있습니다.
- **에이전트** — 검색할지, 어떤 도구를 부를지, 한 번 더 시도할지, 멈출지를 **모델이 그때그때 판단**합니다. 경로가 입력에 따라 달라지는 것이 핵심 차이입니다.

| 축 | 단일 호출 | RAG | 워크플로우 | 에이전트 |
|----|-----------|-----|-----------|----------|
| 경로 결정 | 없음 | 고정(검색→생성) | 고정(분기 포함) | 모델이 동적 판단 |
| 외부 접근 | 없음 | 지식베이스 | 정해진 도구 | 지식베이스 + 도구(가변) |
| 단계 수 | 1 | 2 | N(고정) | 가변(루프) |
| PAIS 모듈 | Model Runtime | Data Indexing + ④ | (애플리케이션) | Agent Builder |

실무에서는 경계가 섞입니다. PAIS의 에이전트는 **관리형 RAG를 내장한 도구 호출 에이전트**에 가깝습니다 — 지식베이스 검색이 에이전트가 쓸 수 있는 하나의 도구로 노출되고, 거기에 MCP 도구가 더해집니다([04](04-mcp-tools.md)). 단순히 "정해진 답을 검색"하면 충분한 경우라면 에이전트가 아니라 ④의 RAG로 푸는 편이 단순하고 예측 가능합니다. 이 판단 기준은 [02 설계 패턴](02-design-patterns.md)에서 다룹니다.

## 1.2 PAIS 2.1 모듈 지형

PAIS 2.1은 여섯 모듈로 이뤄지며, 에이전트는 이들을 함께 씁니다.

### 1.2.1 모델 계층 — Model Gallery · Model Runtime

- **Model Gallery** — 모델 아티팩트를 담는 저장소로, Harbor(OCI 호환 컨테이너 레지스트리)를 기반으로 합니다. NVIDIA NIM·자체 모델 등을 프로젝트 단위로 관리합니다. ([05](05-models-serving.md))
- **Model Runtime** — Gallery의 모델을 추론 엔진으로 실행해 **OpenAI 호환 엔드포인트**로 노출합니다. completion(생성)·embedding(임베딩) 모델을 서빙합니다. 에이전트는 여기서 노출된 completion 엔드포인트를 가져다 씁니다.

### 1.2.2 지식 계층 — Data Indexing and Retrieval

데이터소스를 인덱싱해 **지식베이스**로 묶고, 검색을 제공합니다. 벡터 임베딩은 pgvector 확장을 갖춘 외부 PostgreSQL에 저장됩니다. 에이전트는 지식베이스를 연결해 검색을 수행하며, 검색 자체가 MCP 도구로 노출됩니다(아래 1.2.3). 지식베이스 구성·청크·임베딩의 상세는 ④에 위임합니다.

### 1.2.3 도구 계층 — MCP Servers and Tool Gallery (2.1 신규)

Model Context Protocol(MCP)로 외부 시스템을 도구로 연결합니다. PAIS는 MCP를 세 방향으로 다룹니다 — 에이전트가 외부 MCP 도구를 **호출**하고, 지식베이스 검색을 MCP 도구로 **호스팅**하며, 원격 MCP 서버를 도구로 **등록·승인**합니다. **Tool Gallery**는 이 MCP 서버들을 중앙에서 관리하는 2.1 신규 기능입니다. ([04](04-mcp-tools.md))

### 1.2.4 조립 계층 — Agent Builder

모델 엔드포인트·지시문·지식베이스·도구·세션 정책을 묶어 에이전트를 구성하고, **채팅 완성(chat completion) 엔드포인트**로 노출합니다. 모델과 지식·도구 사이의 상호작용을 오케스트레이션하는 것이 Agent Builder의 역할입니다. ([03](03-agent-builder.md))

### 1.2.5 관측 계층 — Observability (2.1 확장)

추론 엔진·GPU 사용률부터 지식베이스 인덱싱과 에이전트 동작까지 추적합니다. OpenTelemetry 기반 LLM 추적(trace)으로 "사용자 ↔ 모델 ↔ 에이전트 ↔ 지식베이스" 상호작용을 따라갈 수 있습니다. ([07](07-operations.md))

## 1.3 에이전트의 구성요소

Agent Builder에서 에이전트 하나는 다음 구성요소로 이뤄집니다.

| 구성요소 | 무엇을 정하나 | 출처 모듈 |
|----------|--------------|-----------|
| 모델 엔드포인트(completion) | 추론에 쓸 모델 | Model Runtime |
| 지시문(Instructions) | 시스템 프롬프트·행동 정책(선택) | Agent Builder |
| 지식베이스(Knowledge Bases) | 검색 대상(선택) | Data Indexing |
| 도구(Tools) | 호출 가능한 MCP 도구 | MCP / Tool Gallery |
| 세션 파라미터 | 대화 이력 길이·유지 시간(TTL)·요약 전략 | Agent Builder |
| 검색 파라미터 | Top-K·유사도 컷오프·인용 표시 | Agent Builder |

이 구성요소들을 어떻게 채우는지가 [03 Agent Builder로 구축](03-agent-builder.md)의 주제입니다. 핵심은 **에이전트가 코드가 아니라 구성으로 정의된다**는 점입니다 — 위저드로 구성하고, 그 결과를 REST API로 소비하며, 구성 코드를 내보내 형상관리할 수 있습니다.

## 1.4 에이전트는 어디서 도는가

- **실행 기반** — PAIS는 Supervisor의 vSphere Namespace 안에 프로비저닝되는 **VKS(vSphere Kubernetes Service) 클러스터** 위에서 동작합니다. Supervisor에 PAIS용 Kubernetes Operator가 배포되고, 관리 파드(API·UI 백엔드·인덱싱 워커)와 모델 엔드포인트 파드가 VKS 워커 노드에서 실행되며 ESXi 호스트의 GPU에 연결됩니다.
- **노출 형태** — 에이전트는 Model Runtime의 stateless 엔드포인트 위에 **세션·검색·도구 호출을 더한 stateful 계층**으로 동작하며, 채팅 완성 엔드포인트로 호출됩니다. (에이전트가 독립된 장기 실행 파드로 배치되는지, Agent Builder 서비스 안의 세션 계층으로 동작하는지는 공식 문서가 단정하지 않으므로, 배포 단위는 적용 환경에서 확인하시기 바랍니다.)
- **GPU 의존** — 모델 엔드포인트는 GPU(또는 2.1부터 llama.cpp로 CPU)에서 추론합니다. 에이전트 자체는 모델·도구를 오케스트레이션하는 경량 계층이며, 무거운 비용은 모델 추론에서 발생합니다([07](07-operations.md) 비용).

다음 문서에서는 이 구성요소들을 **어떻게 조합할지** — 단일·멀티 에이전트, 도구·지식 연결, 세션 관리 — 를 설계 관점에서 다룹니다.
