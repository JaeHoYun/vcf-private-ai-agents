# 06 — 평가와 가드레일

[← 목차로](../README.md)

에이전트는 출력이 매번 다르고 경로가 동적이라, 일반 소프트웨어처럼 단언적으로 테스트하기 어렵습니다. 이 문서는 PAIS가 평가에 무엇을 제공하고 무엇은 제공하지 않는지를 분명히 한 뒤, 무엇을 어떻게 측정할지, 어떤 실패 모드를 막을지, 가드레일과 사람 개입을 어디에 둘지를 다룹니다.

> 본 문서의 수치·동작은 작성 시점(2026-06) VCF 9.1 / PAIF 9.1 / PAIS 2.1 기준이며, 적용 전 공식 문서로 재확인하시기 바랍니다.

---

## 6.1 평가가 어려운 이유

에이전트의 출력은 결정적이지 않습니다. 같은 입력에도 표현이 달라지고, 검색·도구 호출의 경로가 입력에 따라 바뀝니다. 그래서 "정확히 이 문자열을 반환한다"는 단언적 테스트가 잘 들어맞지 않습니다. 평가는 **단일 통과/실패가 아니라 분포·경향**으로 봐야 하며, 대표 입력 묶음에 대한 반복 측정이 필요합니다.

## 6.2 PAIS가 제공하는 것 — 그리고 제공하지 않는 것

- **Playground(대화형 테스트)** — Agent Builder에 내장된 1차 검증 수단입니다. 사람이 직접 대화하며 검색·도구 호출·거절 동작을 눈으로 확인하고 지시문을 다듬습니다([03 §3.7](03-agent-builder.md)).
- **CI/CD 자동 테스트** — 구성 코드를 형상관리에 두고([03 §3.8](03-agent-builder.md)) 파이프라인에서 대표 입력으로 에이전트를 호출해 회귀를 잡는 패턴이 문서화돼 있습니다.

> **솔직한 경계** — PAIS 2.1에 이름 붙은 전용 평가(eval) 스코어링 프레임워크가 있다는 공식 근거는 확인되지 않습니다. 따라서 정량 평가 묶음(테스트셋·채점 기준·회귀 추적)은 **애플리케이션·CI 계층에서 직접 설계**해야 합니다. Playground는 탐색적 1차 검증, CI는 반복 검증으로 役을 나누십시오.

## 6.3 무엇을 측정하나

대표 입력 묶음을 정해 다음을 반복 측정하십시오.

| 측정 대상 | 무엇을 보나 |
|-----------|-------------|
| 검색 적절성 | 필요한 때 검색하는가, 검색어·Top-K·컷오프가 맞는가 |
| 도구 선택 정확도 | 올바른 도구를 부르는가, 인자가 맞는가 |
| 근거성(환각 여부) | 답이 검색·도구 결과에 근거하는가, 인용이 맞는가 |
| 종단 품질 | 최종 답이 과업을 실제로 해결하는가 |
| 비용·지연 | 토큰·단계 수·TTFT·종단 지연이 허용 범위인가([07](07-operations.md)) |

지시문·검색 파라미터·모델을 바꿀 때마다 이 묶음을 다시 돌려 **회귀**를 확인하십시오. 한 입력에서 좋아진 변경이 다른 입력을 망치는 일이 흔합니다.

## 6.4 실패 모드

에이전트는 단일 호출에 없던 실패를 들여옵니다.

- **환각(근거 없는 단정)** — 검색·도구 결과 없이 그럴듯한 답을 지어냅니다. 인용 표시·근거성 평가·지시문의 "모르면 모른다고" 규칙으로 완화합니다.
- **도구 오선택** — 비슷한 도구를 잘못 고르거나 인자를 틀립니다. 도구 수를 줄이고 이름·설명을 명확히 하면 줄어듭니다([02 §2.3](02-design-patterns.md)).
- **루프** — 같은 검색·도구 호출을 반복하며 끝내지 못합니다. 최대 단계·시간 한계로 종료 조건을 둡니다.
- **권한 초과** — 의도 밖 시스템에 쓰기·전송을 시도합니다. 도구 권한 최소화와 승인 통제로 막습니다([04 §4.7](04-mcp-tools.md)).

## 6.5 가드레일 — PAIS의 한계와 보완

"가드레일"이라는 말은 PAIS 맥락에서 두 가지 인프라 의미로 주로 쓰입니다.

- **리소스 가드레일** — 격리된 vSphere Namespace의 CPU·메모리·GPU 쿼터. 한 워크로드가 자원을 독식하지 못하게 막는 운영 통제입니다.
- **거버넌스 포지셔닝** — 에이전트 루프·도구 사용에 대한 상위 플랫폼 차원의 거버넌스 방향.

> **솔직한 경계** — 콘텐츠 모더레이션·프롬프트 인젝션 방어 같은 **PAIS 내장 콘텐츠 가드레일** 기능은 공식 근거가 확인되지 않습니다. 따라서 출력 안전·입력 검증·민감정보 차단 같은 가드레일은 **애플리케이션 계층에서 설계**해야 합니다 — 입력·출력 필터, 도구 권한 최소화, 승인 통제([04](04-mcp-tools.md)), 지시문의 거절 규칙을 조합하십시오. 통제·정책의 상세는 [⑤ 보안·거버넌스](https://github.com/JaeHoYun/vcf-private-ai-security-governance)에 위임합니다.

## 6.6 휴먼인더루프

되돌리기 어려운 행동(쓰기·전송·결제·구성 변경) 앞에 사람의 확인을 두는 것은 에이전트 안전의 핵심입니다. 그러나 PAIS 2.1에 런타임 응답을 사람이 승인·차단하는 **내장 휴먼인더루프 기능**은 확인되지 않습니다.

- MCP 도구의 "승인"은 관리자가 **도구 등록을 통제**하는 절차이지, 매 호출을 사람이 승인하는 것이 아닙니다([04 §4.4](04-mcp-tools.md)).
- 따라서 위험한 도구 호출에 대한 사람 확인 게이트는 **애플리케이션 계층에서 구현**해야 합니다 — 예컨대 쓰기 도구를 직접 노출하는 대신, 제안을 만들고 사람이 승인하면 실행하는 흐름으로 감쌉니다.

이 경계를 분명히 알고 설계하면, "에이전트가 알아서 다 한다"는 위험한 기대 대신 **통제 가능한 자동화**를 만들 수 있습니다. 다음 문서에서는 이렇게 만든 에이전트를 실제 운영에 올리는 Day-2를 다룹니다.
