# 02 — 에이전트 설계 패턴

[← 목차로](../README.md)

에이전트를 만들 수 있다는 것과 만들어야 한다는 것은 다릅니다. 이 문서는 구현에 들어가기 전 결정할 설계 질문을 다룹니다 — 언제 에이전트로 풀지, 하나로 둘지 여럿으로 나눌지, 도구와 지식을 얼마나 붙일지, 세션을 어떻게 관리할지. PAIS의 구성요소에 매핑되는 실무 기준으로 좁혀 설명합니다.

> 본 문서의 수치·동작은 작성 시점(2026-06) VCF 9.1 / PAIF 9.1 / PAIS 2.1 기준이며, 적용 전 공식 문서로 재확인하시기 바랍니다.

---

## 2.1 언제 에이전트로 푸나

에이전트는 할 수 있는 일이 넓은 만큼 비싸고 예측이 어렵습니다. 동적 경로는 디버깅·평가·비용 통제를 어렵게 만듭니다. 다음을 먼저 따져 **더 단순한 수단으로 충분한지** 확인하십시오.

| 상황 | 권장 수단 |
|------|-----------|
| 정해진 질문에 지식베이스 답변만 필요 | RAG (④) — 에이전트 불필요 |
| 단계가 고정돼 있고 분기가 예측 가능 | 워크플로우(애플리케이션 코드) |
| 무엇을 검색·호출할지 입력마다 달라짐 | 에이전트 |
| 여러 시스템을 조합해 다단계로 처리 | 에이전트 |

**에이전트가 정당해지는 신호** — (1) 입력에 따라 필요한 도구·검색이 달라진다, (2) 한 요청을 처리하려면 중간 결과를 보고 다음 행동을 정해야 한다, (3) 도구 조합의 경우의 수가 많아 코드로 고정하기 어렵다. 이 신호가 약하면 RAG나 워크플로우가 더 안전합니다. 에이전트로 정했다면, 그 에이전트를 관리형 Agent Builder로 만들지 자체 앱으로 직접 오케스트레이션할지는 §2.6에서 정합니다.

## 2.2 단일 에이전트와 멀티 에이전트

- **단일 에이전트** — 하나의 지시문·도구 집합·지식베이스로 한 영역을 담당합니다. 시작점으로 권장합니다. 도구 수가 적고 책임이 분명할 때 디버깅·평가가 쉽습니다.
- **멀티 에이전트** — 영역별 에이전트를 두고 상위에서 라우팅하거나 위임합니다. 도구·지식이 영역마다 크게 다르거나, 한 에이전트의 지시문·도구 목록이 비대해져 모델이 도구 선택을 자주 틀릴 때 고려합니다.

PAIS의 에이전트는 각각 독립된 채팅 완성 엔드포인트로 노출되므로, **한 에이전트를 다른 에이전트의 도구로 부르는 구성**도 가능합니다(MCP 또는 애플리케이션 계층에서 조합). 다만 멀티 에이전트는 지연·비용·실패 지점이 곱으로 늘어납니다. 단일 에이전트의 도구 선택 정확도가 떨어진다는 증거가 있을 때 분할하고, 처음부터 멀티로 시작하지 마십시오.

**분할 기준(실무)**

- 한 에이전트의 도구가 대략 두 자릿수에 이르고 모델이 도구를 자주 잘못 고른다.
- 영역마다 접근 권한·데이터 경계가 달라 한 지시문에 섞으면 권한이 과대해진다([⑤ ID·인증·접근통제](https://github.com/JaeHoYun/vcf-private-ai/blob/main/05-security/docs/03-identity-access.md) 위임).
- 영역별로 다른 모델(크기·비용)이 합리적이다.

## 2.3 도구와 지식 연결 설계

에이전트의 능력은 붙인 도구와 지식으로 정해집니다. 무작정 많이 붙이면 모델의 선택 부담이 커지고 보안 표면이 넓어집니다.

- **지식베이스는 검색 도구다** — PAIS에서 지식베이스 검색은 에이전트가 호출 여부와 검색어를 스스로 정하는 도구로 노출됩니다. 검색이 항상 필요한 단순 Q&A라면 에이전트보다 ④ RAG가 단순합니다.
- **도구는 최소·명확하게** — 에이전트가 자주 쓸 도구만 노출하고, 이름·설명·파라미터를 모델이 헷갈리지 않게 작성합니다. 비슷한 도구가 여럿이면 모델이 잘못 고릅니다.
- **도구는 승인된 것만** — 원격 MCP 서버의 도구는 관리자가 능력·파라미터를 검토해 명시적으로 승인한 것만 에이전트에 노출됩니다([04](04-mcp-tools.md)). 승인은 보안 경계인 동시에 도구 표면을 좁히는 설계 수단입니다.
- **권한 최소화** — 도구가 닿는 시스템의 자격증명·범위는 필요한 최소로 둡니다. 도구 자체가 외부 시스템에 대한 공격 표면이 됩니다([⑤ ID·인증·접근통제](https://github.com/JaeHoYun/vcf-private-ai/blob/main/05-security/docs/03-identity-access.md) 위임).

## 2.4 세션과 컨텍스트 설계

에이전트는 대화를 이어가는 stateful 워크로드입니다. Agent Builder의 **세션 파라미터**로 다음을 정합니다.

- **대화 이력 길이** — 컨텍스트에 유지할 과거 대화의 양. 길수록 맥락이 풍부하지만 토큰·비용·지연이 늘고, 모델 컨텍스트 한계에 부딪힙니다.
- **유지 시간(TTL)** — 세션을 얼마나 살려둘지.
- **요약 전략** — 이력이 길어질 때 과거를 요약해 압축하는 방식. 긴 대화에서 토큰 폭증을 막습니다.

**검색 파라미터** — Top-K(가져올 문서 수)·유사도 컷오프(0~1)·인용 표시. Top-K를 키우면 맥락은 늘지만 잡음과 토큰도 늘고, 컷오프를 높이면 정밀해지지만 누락이 늘 수 있습니다. 기본값에서 시작해 평가([06](06-evaluation-guardrails.md))로 조정하십시오.

**도구 호출을 지원하지 않는 모델** — 일부 모델은 네이티브 도구 호출(tool calling)을 지원하지 않습니다. PAIS 2.1은 이런 모델로도 에이전트를 구성할 수 있는 옵션을 제공하지만(별도 구성 필요), 도구를 적극적으로 쓰는 에이전트라면 도구 호출을 지원하는 모델을 선택하는 편이 단순합니다([05](05-models-serving.md) 모델 선택).

## 2.5 실패를 전제한 설계

에이전트는 새로운 실패 모드를 들여옵니다 — 도구를 잘못 고르거나, 같은 행동을 반복하는 루프에 빠지거나, 근거 없이 단정(환각)합니다. 설계 단계에서 다음을 미리 정하십시오.

- **종료 조건** — 무한 루프를 막을 최대 단계·시간 한계.
- **도구 오류 처리** — 도구가 실패하거나 비정상 응답을 줄 때 에이전트가 어떻게 대응하는가.
- **사람 개입 지점** — 되돌리기 어려운 행동(쓰기·전송·결제) 앞에 사람의 확인을 둘지. PAIS에 내장된 휴먼인더루프 기능은 확인되지 않으므로, 이런 게이트는 **애플리케이션 계층에서 설계**해야 합니다([06](06-evaluation-guardrails.md)).

이 실패 모드와 평가·가드레일은 [06 평가와 가드레일](06-evaluation-guardrails.md)에서 자세히 다룹니다.

## 2.6 관리형 Agent Builder와 자체 앱

에이전트로 풀기로 했다면, 그 에이전트를 **무엇으로 만들지**가 다음 갈림길입니다. PAIS 위에서는 두 경로가 있습니다.

- **관리형 — Agent Builder** — 모델·지시문·지식베이스·도구·세션을 UI로 묶고, 검색·도구 호출·세션을 PAIS가 얹어 채팅 완성 엔드포인트로 노출합니다([03](03-agent-builder.md)). 구성이 빠르고 운영·관측이 플랫폼에 통합됩니다.
- **자체 앱 — 직접 오케스트레이션** — 계획·분기·상태를 잇는 오케스트레이션 루프를 애플리케이션 코드나 외부 프레임워크(예: LangGraph·LlamaIndex)로 직접 짜고, PAIS의 **모델 엔드포인트(OpenAI 호환)만 백엔드로 소비**합니다([05 §5.1](05-models-serving.md)). Model Runtime이 OpenAI 규약을 따르므로, 기존 프레임워크의 모델 백엔드 주소(base URL)만 PAIS로 바꿔 그대로 쓸 수 있습니다.

두 경로는 배타적이지 않습니다 — 자체 앱이 Agent Builder 에이전트 엔드포인트를 한 단계로 호출하는 **하이브리드**도 가능합니다(에이전트도 채팅 완성 엔드포인트이므로).

**트레이드오프**

| 기준 | 관리형 Agent Builder | 자체 앱(직접 오케스트레이션) |
|------|----------------------|------------------------------|
| 구성 속도 | 빠름(UI 구성) | 느림(코드·프레임워크) |
| 오케스트레이션 제어 | 관리형 내부 루프 | 완전 제어(복잡 계획·분기·상태) |
| 이식성·종속 | PAIS 구성에 묶임 | 프레임워크 이식성 높음, 엔드포인트만 의존 |
| 운영·관측 | 플랫폼 통합([07](07-operations.md)) | 앱이 직접 구성 |
| 평가·가드레일 | 앱 계층 설계([06](06-evaluation-guardrails.md)) | 앱 계층 설계(동일) |

**고르는 기준**

- **관리형으로 충분** — 검색·도구 호출·세션을 잇는 표준적 에이전트이고, 빠른 구성과 플랫폼 통합 운영이 우선일 때 Agent Builder.
- **자체 앱이 합리적** — 다단계 계획·조건 분기·외부 상태·멀티에이전트 조율처럼 오케스트레이션을 세밀하게 제어해야 하거나, 기존 프레임워크 자산을 재사용·이식해야 할 때.
- **하이브리드** — 큰 흐름은 자체 앱이 쥐되, 잘 정의된 하위 에이전트는 Agent Builder로 만들어 엔드포인트로 소비.

> **경계** — Agent Builder가 내부적으로 어떤 추론 전략(예: 함수 호출 루프)을 쓰는지, 단계 상한·계획을 사용자가 어디까지 제어할 수 있는지는 공식 문서로 확인하십시오. 세밀한 추론 제어가 핵심 요구라면 자체 앱 경로가 안전합니다.

관리형 경로는 다음 문서에서 Agent Builder로 구성합니다. 자체 앱이든 관리형이든, 완성된 에이전트는 채팅 완성 엔드포인트로 소비합니다([03 §3.8](03-agent-builder.md)).
