# 08 — 산업 유스케이스: 결과·KPI로 보는 에이전트

[← 목차로](../README.md)

앞의 00–07이 에이전트를 **어떻게 만들고 운영하는가**를 다뤘다면, 이 문서는 한 걸음 물러서서 **무엇을·왜 만드는가**를 다룹니다. 도입을 검토하는 입장에서 가장 먼저 떠오르는 질문 — "그래서 이걸로 무슨 일을 하고, 어떤 지표가 좋아지며, 왜 하필 에이전트인가" — 에 산업 시나리오로 답합니다.

여기 실리는 사례는 특정 고객이 아니라 **일반화한 산업 아키타입**입니다. 실제 기업·조직명을 결부하지 않으며, 자사 환경에 맞춰 재해석하는 출발점으로 쓰시기 바랍니다.

> 본 문서의 KPI는 **측정해야 할 지표**이지 보장된 수치가 아닙니다. 실제 효과는 데이터 품질·업무 흐름·도입 범위에 따라 달라지므로, 도입 전후를 직접 계측해 검증하시기 바랍니다. 수치·동작은 작성 시점(2026-06) VCF 9.1 / PAIF 9.1 / PAIS 2.1 기준입니다.

---

## 8.1 이 문서를 읽는 법

각 유스케이스는 기능 목록이 아니라 **결과 중심**으로 적었습니다. "무슨 기능이 있다"가 아니라 "어떤 문제를, 어떤 방식으로 풀어, 무엇이 좋아지는가"의 순서입니다. 기술 구현의 세부는 이 가이드의 해당 장(章)과 형제 가이드로 링크합니다.

이 문서는 **제조 산업을 깊게** 먼저 다룹니다. 여기서 확정한 형식(§8.2)을 다른 산업(유통·방산·금융)으로 같은 틀로 확장하는 것이 다음 단계입니다(§8.6).

## 8.2 유스케이스 공통 형식 (템플릿)

모든 유스케이스를 아래 여섯 항목으로 기술합니다. 새 산업·사례를 추가할 때도 같은 틀을 따릅니다.

| 항목 | 무엇을 적는가 |
|------|---------------|
| **상황** | 지금 어떻게 일하고, 무엇이 병목인가 (도입 전 모습) |
| **에이전트가 하는 일** | 입력 → 처리 → 출력. 사람이 하던 어느 단계를 어떻게 바꾸는가 |
| **왜 이 방식인가** | 왜 Private AI인가 / 왜 파인튜닝이 아니라 RAG인가 / 왜 단순 질의응답이 아니라 에이전트인가 |
| **측정 지표(KPI)** | 도입 효과를 가늠할 계측 대상. 단언된 수치가 아니라 "무엇을 재는가" |
| **구성 매핑** | 어느 가이드·모듈로 구현하는가 (④ RAG · 본 가이드 Agent Builder/MCP · ⑤ 보안 · ⑥ 사이징) |
| **전제·주의** | 데이터 준비, 가드레일, 휴먼인더루프, 리스크 |

## 8.3 제조 — 기술문서 Q&A 에이전트

**상황** — 현장 엔지니어는 장비 매뉴얼·도면·사내 규격·과거 작업지시서가 여러 시스템에 흩어진 상태에서 답을 찾습니다. 문서가 수천 페이지에 이르고 개정도 잦아, 정확한 근거를 찾는 데 시간이 들고 숙련자에게 문의가 몰립니다.

**에이전트가 하는 일** — 자연어 질문("이 모델 토크 규격과 개정 이력")을 받아, 사내 문서에서 관련 대목을 검색해 **근거 문서와 함께** 답합니다. 후속 질문의 맥락을 세션으로 유지하고, 필요하면 도면·규격 등 출처 링크를 함께 제시합니다.

**왜 이 방식인가**
- **Private AI** — 도면·규격은 사외 반출이 불가한 자산이므로 사내 추론이 전제입니다.
- **파인튜닝이 아니라 RAG** — 문서가 자주 개정되고 **출처 추적**이 중요합니다. 모델에 암기시키는 대신 최신 문서를 검색해 근거를 대는 방식이 맞습니다([01 §1.1](01-foundations.md) 에이전트 vs RAG 경계 참조).
- **단순 검색이 아니라 에이전트** — 후속 질문·맥락 유지·여러 문서 종합이 필요해, 단발 검색보다 세션을 가진 에이전트가 적합합니다.

**측정 지표(KPI)** — 문서 탐색 평균 소요 시간, 1차(자가) 해결률, 숙련자 문의 건수, 근거 제시 정확도(출처가 실제로 답을 뒷받침하는 비율). 도입 전후를 같은 기준으로 계측해 비교합니다.

**구성 매핑** — 검색·인덱싱은 [④ RAG 레퍼런스](https://github.com/JaeHoYun/vcf-rag-reference-architecture), 에이전트·세션은 [03 Agent Builder](03-agent-builder.md), 문서 접근통제는 [⑤ 보안·거버넌스](https://github.com/JaeHoYun/vcf-private-ai-security-governance).

**전제·주의** — 문서 ACL(접근권한)이 검색 결과에 반영되어야 합니다(권한 없는 문서가 답에 새지 않도록). 근거 없는 단정(환각)을 막는 가드레일과, 안전·규격 관련 답변의 휴먼 확인 경계가 필요합니다([06 평가와 가드레일](06-evaluation-guardrails.md)).

## 8.4 제조 — 품질 이상 분석 리포트 보조

**상황** — 라인에서 불량·이상이 발생하면 담당자가 과거 유사 사례, 추정 원인, 표준 조치를 여러 시스템에서 찾아 리포트를 작성합니다. 사례 검색과 문서화에 시간이 들고, 작성 품질이 담당자 숙련도에 좌우됩니다.

**에이전트가 하는 일** — 이상 내용을 입력받아, 과거 유사 사례와 원인·조치 이력을 검색하고, 품질 시스템의 관련 데이터를 **도구로 조회**해, 원인 가설과 조치안을 담은 **리포트 초안**을 만듭니다. 담당자는 초안을 검토·수정해 확정합니다.

**왜 이 방식인가**
- **에이전트(도구 호출)** — 문서 검색만으로는 부족하고, 품질·이력 시스템을 조회하는 단계가 끼어듭니다. [04 MCP 도구](04-mcp-tools.md)로 사내 시스템을 표준 인터페이스로 붙여 모델이 호출하게 합니다.
- **RAG** — 과거 사례·표준 조치는 검색으로 근거를 끌어옵니다.
- **휴먼인더루프 전제** — 품질 판단은 사람이 확정합니다. 에이전트는 초안 작성까지를 맡아 시간을 줄이는 역할입니다.

**측정 지표(KPI)** — 리포트 작성 평균 시간, 유사 사례 검색 적중률, 초안 채택률(사람이 크게 고치지 않고 쓰는 비율), 재발 분석 누락 감소. 도입 전후를 같은 기준으로 계측해 비교합니다.

**구성 매핑** — 사례 검색은 [④ RAG](https://github.com/JaeHoYun/vcf-rag-reference-architecture), 시스템 조회는 [04 MCP 도구](04-mcp-tools.md), 초안 생성·세션은 [03 Agent Builder](03-agent-builder.md), 판단 경계는 [06 가드레일](06-evaluation-guardrails.md).

**전제·주의** — 도구 호출 권한과 감사 로그가 명확해야 합니다(어떤 데이터를 누가·왜 조회했는가). 초안을 사실로 오인하지 않도록 "초안·검토 필요" 표시와 휴먼 확정 단계를 강제합니다.

## 8.5 제조 — 설비 보전 매뉴얼 RAG

**상황** — 설비 보전 기술자는 정비 절차, 부품 규격, 고장 이력, 안전 지침을 빠르게 확인해야 하지만, 현장에서 두꺼운 매뉴얼과 분산된 이력 시스템을 뒤지는 데 시간이 듭니다. 야간·교대 상황에서 숙련자 부재 시 대응이 지연됩니다.

**에이전트가 하는 일** — "이 설비 OO 경보의 점검 절차"를 자연어로 묻고, 매뉴얼의 해당 절차와 안전 지침을 근거와 함께 받습니다. 필요하면 보전 이력 시스템을 조회해 직전 정비 내역을 함께 제시합니다.

**왜 이 방식인가**
- **현장 즉시성** — 숙련자에게 의존하던 절차 확인을 셀프로 바꿔, 교대·야간 대응 격차를 줄입니다.
- **RAG + 도구** — 절차·지침은 문서 검색(RAG), 이력은 보전 시스템 조회(MCP 도구)로 결합합니다.
- **안전 우선** — 안전 관련 절차는 근거 문서를 반드시 함께 제시하고, 임의 생성 답변을 억제합니다.

**측정 지표(KPI)** — 절차 확인 소요 시간, 교대·야간 대응 지연, 매뉴얼 미준수로 인한 재작업 빈도, 안전 지침 근거 제시율.

**구성 매핑** — [④ RAG](https://github.com/JaeHoYun/vcf-rag-reference-architecture) · [04 MCP 도구](04-mcp-tools.md) · [03 Agent Builder](03-agent-builder.md) · 안전·가드레일은 [06](06-evaluation-guardrails.md).

**전제·주의** — 안전·규격 영역은 환각이 곧 사고 위험입니다. 근거 없는 답변 차단, 출처 강제, 위험도 높은 절차의 휴먼 확인을 기본값으로 둡니다. 매뉴얼 개정이 검색 인덱스에 제때 반영되는 운영 절차가 필요합니다([07 운영](07-operations.md)).

## 8.6 다음 산업으로 확장

위 세 사례는 §8.2의 공통 형식을 제조에 적용한 것입니다. 같은 틀로 다른 산업을 확장할 때의 출발점을 적어 둡니다(상세는 후속 작업).

- **유통** — 사내 상품·정책 지식 Q&A, 점포 운영 매뉴얼 보조, 클레임 대응 초안.
- **방산** — 규격·절차 준수 확인, 망분리(에어갭) 환경의 문서 Q&A([05 모델과 서빙](05-models-serving.md)의 에어갭(AMT) 연계).
- **금융** — 내규·컴플라이언스 질의 보조, 심사 근거 문서 검색(고위험 판단은 휴먼 확정 강제).

각 산업도 **상황 → 하는 일 → 왜 이 방식 → KPI → 구성 매핑 → 전제·주의**의 여섯 항목으로 기술해, 결과 중심·근거 기반·실명 비결부 원칙을 동일하게 따릅니다.
